Your project's config finally gets an interface.

visual-config is a visual control surface for JavaScript/TypeScript project configuration (dependencies, scripts, TypeScript, publishing) that lives on top of the files you already have. Every change is a reviewed diff. Everything is reversible. Your files stay the source of truth.

Four kinds of people keep ending up here.

Config is the operating system of a project, and everyone touches it differently. Pick your seat.

You ship apps

Stop hand-editing a dozen JSON files

One command opens your whole project's config as a UI. Any repo, no dev server, no setup.

  • Outdated deps as facts, with Upgrade all behind one reviewed diff
  • Install by selecting from a catalog, so an npx typo can't execute a random package
  • Scripts become buttons with live output
  • Undo anything, from the journal
You run agents

Guardrails instead of free-typed shell

visual-config mcp hands your agent the same operations you get as buttons: validated, previewable, reversible.

  • Every mutation returns a structured diff, not a side effect
  • Format- and comment-preserving writes, enforced scope
  • Supervised mode: the agent's diff lands in your Diff Sheet
  • MCP resources expose the project model read-only
You build tools

Ship your ecosystem's vertical as a plugin

The built-ins are plugins with zero special privileges. Your framework or linter can have a first-class panel.

  • definePlugin from @apostel/visual-config-kit
  • Operations you register appear in the UI and as MCP tools, free
  • Plugins can't touch the filesystem; they return Changes
  • Auto-discovered from the project's dependencies
You have taste

Opinions are packs, not defaults

The base states only verifiable facts. Recommendations come from opinion packs users install, always labeled with your name.

  • “Enable strict” shows as your advice, attributed
  • Users stack packs, see where they disagree, pick
  • Verified vs community badges
  • Local-first, no account, no telemetry

What works today, v0.5.x.

Alpha, published to npm and green in CI. Dependency health, a config editor, tool scaffolding, a one-click Biome swap, changelog reading, and an MCP server all ship today — the vision docs describe what's still ahead (monorepos, IDE panels).

●npx-able: boots a local daemon, opens the React UI on any repo
●Overview: inline-editable name, version, description
●Dependency health: outdated, vulnerabilities (npm advisory DB) and deprecations with the maintainer's suggested alternative
●Changelog & bump safety: read release notes between versions, breaking changes cross-referenced against how your code uses the package
●Catalog: search npm, install by selecting, never by typing
●Config editor: Biome / Prettier / ESLint / oxlint / tsconfig as guided forms with inline docs; read-only static views of next.config / vite.config
●Setup & swaps: scaffold a tool in one step, or one-click Switch to Biome (replaces ESLint + Prettier) — fully reversible
●Scripts: run as buttons with streamed output and Stop
●Diff Sheet: every mutation previewed, confirmed, journaled, undoable
●MCP server: every operation as an agent tool, read-only resources, and an in-session app UI (MCP Apps / SEP-1865)
●Plugins: built-ins load as a plugin; third parties add operations and detectors
●Opinion packs: attributed recommendations, zero baked-in taste
●Format-preserving writes: minimal diffs, comments survive, golden-tested
●Enforced scope: operations declare what they may write; the engine rejects the rest

Your agent shouldn't free-type rm-adjacent shell into your config.

Every button in the UI is also an MCP tool. Agents plan the same Operations, get back the same structured diff, and write through the same validated, reversible engine. As agents write more of your code, the config layer is exactly where you want them constrained and observable.

The roadmap: sequence and scope, not a schedule.

Each phase is independently useful and builds on the same core. Full detail with exit criteria in docs/ROADMAP.md.

Milestone 0: the end-to-end spine shipped

Core → daemon → birpc → React UI → Change → journal, proven on the smallest feature. Runs on any repo.

Phase 0: foundations shipped

The format- and comment-preserving write layer (golden-tested), the Operation→Change→undo engine with enforced scope, and the plugin host the built-ins themselves run on.

Phase 1: the npx browser tool on npm vulns pending

package.json viewer, scripts as buttons, the catalog, the Diff Sheet, outdated-dependency facts with safe upgrades. Vulnerability data (npm audit / OSV) is the open item.

Phase 2: understand & improve config in progress

TypeScript panel and the attributed opinions system are in. Next: changelog-driven, code-aware bump safety; migrations backed by codemods or maintainer-authored agent skills; guided framework config; publish-readiness checks.

Phase 3: the MCP server shipped early

Landed ahead of sequence because the Operation abstraction made it nearly free. The clearest moat: nobody else exposes reversible config mutations to agents.

Phase 4: open the plugin API started

definePlugin, npm auto-discovery, and a worked opinion pack exist. To come: stabilized contribution surface, declarative config editors, the marketplace.

Phase 5: the hard, flashy swaps planned

Biome ⇄ ESLint+Prettier as one previewed, reversible transaction, honest about lossy rule translation. Only after the diff/undo machinery has earned trust.

Phase 6: IDE panels planned

VS Code webview panel first, JetBrains via JCEF, Zed through MCP. Plus the honest, IDE-native “cleaner workspace” toggle: decluttering, never hiding state.

The invariants are the product.

Six rules the codebase enforces, not aspirations.

Files are the only source of truth

No shadow store, no lock-in layer. We read and write the exact files git already tracks. Uninstalling leaves your project byte-for-byte as it was.

No write without a confirmed Change

plan() never writes. One engine method applies changes, and every operation declares the files it may touch; the engine rejects the rest.

Format-preserving, or an explicit refusal

Minimal diffs a careful human would produce. The moment a tool clobbers your comments or reorders your keys, trust is gone, so that layer is golden-tested.

Reversible by default

Every applied change lands in a journal with its inverse. Undo is a first-class citizen for human and agent actions alike.

Facts vs. opinions

The base states only what's verifiable: outdated, vulnerable, non-default. Taste arrives via opinion packs you install, always attributed to their author.

Local-first, no account, no telemetry

It runs on your machine against your files. That's the whole data flow.
